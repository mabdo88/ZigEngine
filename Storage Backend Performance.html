Top Finding
TimeSeries and Columnar are tied for fastest at the time-window aggregation query — both at ~12.4µs median latency. This is the most common digital twin dashboard query, and they win it together by a clear margin over the baseline row-oriented stores.
Surprise Finding
Hierarchical is mid-pack, not slow. Despite maintaining a parent-child zone tree, the 24h average query lands at 14.6µs — within 18% of the winners. The tree traversal overhead is real but small for this query shape.
Memory Red Flag
RingBuffer uses 14x more memory than the column-oriented backends (241 KB vs 10 KB for the same 500 readings). It's fast but it's a memory hog — exactly as the design predicts.

Deep Dive: AoS vs SoA

The two baseline layouts. Why does DOD win on paper but tie here?

AoS — Array of Structs
16.5 µs
vs
SoA — Struct of Arrays
16.2 µs
2% faster

AoS Layout

// One struct = one reading
[sensor=0, ts=0, val=21.0]
[sensor=0, ts=1, val=21.1]
[sensor=0, ts=2, val=21.2]
...
// Cache line: 16 bytes per reading
// Scanning "all values" pulls ts+id waste

Why it's slow: To compute AVG(value), you fetch every reading's full struct, including timestamp and sensor_id you don't need.

SoA Layout

// Separate arrays per field
sensor_ids = [0, 0, 0, 0, ...]
timestamps = [0, 1, 2, 3, ...]
values     = [21.0, 21.1, 21.2, ...]

// Cache line: tight on values only

Why it should be fast: AVG only touches the values array. Adjacent values stay in cache. SIMD vectorization is trivial.

Why SoA Only Wins By 2%
At 500 readings, everything fits in L1 cache. Both layouts pull data into cache equally fast. SoA's advantage scales with data size — once you exceed L1 (typically 32-64 KB), SoA starts pulling ahead because it only fetches the field it needs. This benchmark is too small to show SoA's true strength.

Deep Dive: Why TimeSeries Wins

TimeSeries stores readings sorted by timestamp. For time-window queries, that's the perfect layout.

The Layout (Append-Only)

// Index by (sensor_id, timestamp)
[s0, ts=1000, v=21.0]
[s0, ts=1060, v=21.1]
[s0, ts=1120, v=21.2]
[s0, ts=1180, v=21.3]
[s1, ts=1000, v=19.5]
[s1, ts=1060, v=19.6]
...

// "Avg for s0 last 24h"
// = scan contiguous block

What The Benchmark Proves

Contiguous reads — once you know s0's block range, the average computation is a tight loop with no branching and no random access.
Out-of-order inserts still work — tests #20 and #33 confirm reads return sorted data even after lazy sorting kicks in.
Memory at parity with AoS — 16,752 bytes vs AoS's 16,752. The time index costs almost nothing at this size.
But: p99 is only 2.16× median (26.8 µs vs 12.4 µs) — decent tail latency, not exceptional.
Where It Wins In Production
This benchmark is a single-sensor, single-window query. In production, TimeSeries's real wins come from time-range scans that touch millions of rows — sorted-by-time means sequential I/O, which is 100x faster than random I/O on spinning disks and still 5-10x faster on SSDs. This benchmark can't show that advantage at 500 rows.

Maintaining a zone tree costs more than AoS/SoA — but less than you'd expect.

The Tree-Building Cost
Each insert requires maintaining parent-child links and updating zone indexes. That's 3-4 pointer operations per write vs 1 for AoS. Read performance is comparable because the tree shape is queried at most once per call, then the sensor scan dominates.
When Hierarchical Wins Big
For queries like "all sensors on floor 3" or "siblings of room 201", Hierarchical is the only backend with an O(log n) or O(1) answer. The other backends would scan everything. That test isn't in this benchmark suite yet.

RingBuffer: Fast But Greedy

2nd fastest on latency, but the memory story is brutal at small scale.

22.8×
more memory than the columnar winners (for the same 500 readings)
Why It's So Memory-Heavy
RingBuffer pre-allocates a fixed-size slot array per sensor entity. For 10 sensors with capacity 100 readings each = 1000 slots reserved at startup, regardless of how many are filled. At 500 readings, half the slots are empty but committed.
When It Wins Anyway
At full capacity (every slot used), RingBuffer's per-reading memory approaches AoS/SoA. For a stable sensor population where buffer size is well-tuned to expected throughput, the latency win (14.0 vs 16.5 µs) becomes worth the RAM cost.
Test #45 Confirms Eviction Works
eviction keeps only latest N readings — RingBuffer correctly drops old data when full. This is what makes it suitable for the "real-time dashboard" tier in the spec's tiered architecture.

What This Means For The Cost Model

Translating microseconds into dollars — and what the report should actually say.

At This Scale, Choice Doesn't Matter For Cost
500 readings × 10 sensors is a tiny dataset. All backends handle it in single-digit milliseconds per query. At this scale, the cost difference is dominated by storage tier pricing, not query compute. Choose based on the future scale, not the current scale.
What The Report Should Recommend
For a 10-sensor project: any backend works. For a 10k-sensor project: TimeSeries or Columnar based on query shape. For 1M+ sensors: tiered architecture is mandatory, single-backend breaks.

Total: 31B raw readings. Storage is the constraint, not query speed. Recommendation: Edge feature extraction + RingBuffer for last 24h raw + TimeSeries for features + Lake for incident snapshots. No single backend fits.

What This Benchmark Doesn't Tell Us

Honest list of gaps the team should close before trusting the report.

Missing: Data-Volume Scaling
Iterations scale from 100 to 100k, but dataset stays at 500 readings. We have no idea how TimeSeries or Columnar perform at 10M, 100M, or 1B readings. That's where the real production difference lives.
Missing: Multi-Sensor Aggregation
The benchmark is single-sensor. "Average across all sensors on floor 3" is a different query pattern entirely and likely favors Columnar by a much wider margin.
Missing: Write Throughput
No insert performance numbers. Ingest is half the problem and we don't have comparable numbers. RingBuffer should crush here, TimeSeries should also win, AoS likely loses.
Missing: Real Workloads
Dataset is uniform synthetic. No failure modes (dropouts, duplicates, out-of-order arrival percentages from Section 15.6). Real-world data is messier and differences may be larger.
Next Steps
Add a scale_data() benchmark that varies sensor count (10, 100, 1k, 10k, 100k) at fixed iterations. Add a write_throughput() benchmark for ingest. Add a multi-sensor aggregation variant of query_avg_window. With those four additions, the report becomes trustworthy for real deployment decisions.
